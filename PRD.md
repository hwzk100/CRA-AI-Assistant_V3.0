## 1. 产品概述

### 1.1 产品简介

CRA AI Assistant V3.0是一款专为临床研究助理（CRA）设计的智能桌面应用程序。通过集成智谱GLM-4大语言模型，自动化处理临床试验方案文档和受试者医疗记录，提取关键信息并生成符合GCP标准的Excel Tracker报告，显著提升临床试验数据管理效率。

### 1.2 产品定位

- **目标用户**：临床研究助理(CRA)、临床试验数据管理员、医学监查员
- **使用场景**：临床试验现场监查、数据核对、报告生成
- **核心价值**：将原本需要数小时的人工阅读和数据录入工作，缩短至数分钟完成

### 1.3 产品愿景

成为临床试验行业最专业、最智能的AI辅助工具，帮助临床研究工作者从繁琐的文档处理工作中解放出来，专注于更有价值的临床数据质量把控和受试者安全监护工作。

## 3. 功能需求

### 3.1 核心功能模块

#### 3.1.1 存储区A - 方案文件管理

**功能描述**：提供临床试验方案文档的上传、管理和文本提取功能。

**详细需求**：

| 编号 | 功能项 | 优先级 | 详细说明 |
|------|-------|-------|---------|
| F1-1 | 文件上传 | P0 | 支持PDF、图片(PNG/JPG/JPEG/GIF/BMP/TIFF/WEBP)、ZIP压缩包 |
| F1-2 | 批量处理 | P1 | 支持ZIP包批量上传，自动解压并列出内部文件清单 |
| F1-3 | 文件管理 | P1 | 查看已上传文件列表，支持删除操作 |
| F1-4 | PDF文本提取 | P0 | 使用pdf-parse库直接提取PDF文档中嵌入的文本层（非OCR） |
| F1-5 | 图片OCR识别 | P0 | 使用Tesseract.js对图片进行光学字符识别(OCR)，支持多语言 |
| F1-6 | 处理状态显示 | P1 | 显示文件处理进度：待处理→验证中→PDF解析/OCR识别中→已完成 |
| F1-7 | 错误处理 | P1 | 处理失败时显示明确错误信息，支持重试 |

**文件处理技术说明**：
- **PDF文件**：使用`pdf-parse`库直接提取PDF文档中的文本内容，适用于包含可选择/可复制文本的PDF文档
- **图片文件**：使用`Tesseract.js`进行OCR（光学字符识别），将图片中的文字转换为可编辑文本，适用于扫描件或照片
- **压缩包**：使用`adm-zip`库解压并列出文件清单（暂不支持自动处理压缩包内的文件）

**验收标准**：
- 单个文件大小不超过50MB
- **PDF文本提取**：成功率≥95%（针对包含文本层的PDF文档）
- **图片OCR识别**：准确率≥90%（针对清晰打印文档）
- OCR支持语言：简体中文、繁体中文、英文、日文、韩文

#### 3.1.2 工作表1 - 入排标准核对

**功能描述**：从方案文件中自动提取入组标准和排除标准，支持人工编辑和核对。

**详细需求**：

| 编号 | 功能项 | 优先级 | 详细说明 |
|------|-------|-------|---------|
| F2-1 | AI自动提取 | P0 | 使用GLM-4模型从方案内容中提取入排标准 |
| F2-2 | 标准编号 | P1 | 自动识别或生成标准编号(1、2、3...) |
| F2-3 | 标准分类 | P2 | 识别标准的分类(如人口学、医学、实验室检查等) |
| F2-4 | 手动编辑 | P0 | 支持添加、修改、删除入排标准条目 |
| F2-5 | 搜索筛选 | P1 | 支持关键词搜索、分类筛选 |
| F2-6 | 标记功能 | P1 | 支持标记重要标准、添加备注 |
| F2-7 | 导出功能 | P0 | 导出为Excel工作表"入排标准核对" |

**数据结构**：
```typescript
interface InclusionCriteria {
  id: string;
  number: string;          // 编号：1、2、3...
  description: string;     // 完整描述
  category?: string;       // 分类
  notes?: string;          // 用户备注
}

interface ExclusionCriteria {
  id: string;
  number: string;
  description: string;
  category?: string;
  notes?: string;
}
```

**验收标准**：
- AI提取准确率≥85%（针对结构化方案文档）
- 标准完整性≥95%（不遗漏主要标准）
- 支持一次提取最多100条入排标准

#### 3.1.3 工作表2 - 访视时间核对表

**功能描述**：从方案文件中提取访视计划，包括访视时间窗口、程序项目和评估项目。

**详细需求**：

| 编号 | 功能项 | 优先级 | 详细说明 |
|------|-------|-------|---------|
| F3-1 | AI自动提取 | P0 | 从方案流程表/访视计划表中提取访视信息 |
| F3-2 | 访视信息 | P0 | 访视编号、访视名称、时间窗口(开始/结束) |
| F3-3 | 程序项目 | P1 | 提取各访视的程序项目，最多10个主要项目 |
| F3-4 | 评估项目 | P1 | 提取各访视的评估项目，最多10个主要项目 |
| F3-5 | 项目分类 | P1 | 自动识别程序分类(筛选/治疗/随访)和评估类型(生命体征/实验室/心电图等) |
| F3-6 | 手动编辑 | P0 | 支持添加、修改、删除访视和项目 |
| F3-7 | 拖拽排序 | P2 | 支持访visually拖拽调整顺序 |
| F3-8 | 导出功能 | P0 | 导出为Excel工作表"访视时间核对表" |

**数据结构**：
```typescript
interface VisitSchedule {
  id: string;
  visitNumber: string;        // 访视编号：1、2、-1、1+1等
  visitName: string;          // 访视名称
  windowStart: string;        // 窗口开始：Day -28
  windowEnd: string;          // 窗口结束：Day -1
  procedures: Procedure[];    // 程序项目
  assessments: Assessment[];  // 评估项目
}

interface Procedure {
  name: string;
  category: 'screening' | 'treatment' | 'follow-up' | 'other';
  timing: string;
  required?: boolean;
}

interface Assessment {
  name: string;
  type: 'vital' | 'lab' | 'ecg' | 'imaging' | 'questionnaire' | 'other';
  timing: string;
  required?: boolean;
}
```

**验收标准**：
- AI提取准确率≥80%（针对表格化访视计划）
- 访vis时间窗口提取准确率≥90%
- 每个访视最多提取10个程序+10个评估项目

## 6. 用户流程

### 6.1 主要用户旅程

#### 场景1：提取入排标准

```
1. 用户打开应用
2. 点击"添加文件"上传方案PDF
3. 系统自动提取PDF文本
4. 切换到"工作表1"标签
5. 点击"提取入排标准"按钮
6. AI分析方案内容
7. 展示提取的入排标准列表
8. 用户查看并编辑(可选)
9. 数据自动保存
```

#### 场景2：生成完整Tracker报告

```
1. 上传方案文件到存储区A
2. 提取入排标准(工作表1)
3. 提取访视计划(工作表2)
4. 上传受试者病历到存储区B(如需要)
5. 识别用药信息(工作表3)
6. 检查和编辑所有工作表数据
7. 点击"导出Tracker"按钮
8. 选择保存位置
9. 生成Excel文件
10. 打开Excel检查结果
```

### 8.2 提示词工程

产品针对不同功能设计了专门的提示词模板：

1. **入排标准提取**：结构化提取入组和排除标准
2. **访视计划提取**：从流程表中提取访vis信息和项目清单
3. **用药信息识别**：从病历中识别结构化用药信息
4. **受试者信息提取**：提取受试者编号、访视时间等
